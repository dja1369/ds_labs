# AGENDA.md — ds_labs 프로젝트 구현 요구사항 정리

> 이 문서는 브레인스토밍을 통해 확정된 요구사항과 결정 사항을 정리한 핸드오프 문서입니다.
> 시스템/템플릿의 상세 구현 설계는 별도 모델이 담당하며, 이 문서는 그 설계 작업의 입력(컨텍스트)으로 사용됩니다.
> 연구 주제(안건) 후보 리스트는 별도 워크플로우로 병행 진행 중이며, 이 문서의 범위에 포함되지 않습니다.

## 프로젝트 개요

- **이름**: ds_labs
- **성격**: 컴퓨터/인프라 관련 실험을 재현 가능한 형태로 수행하고, 결과를 GitHub Pages에 공개하는 **포트폴리오 성격의 실험 랩** (외부 공개 염두)
- **목적**: 실험과 학습. 특정 "안건(연구 질문)"을 정하면, 템플릿 기반으로 환경을 구성해 실제로 부하/동작을 테스트하고, 그 결과를 이미지 포함 문서로 남기고, 다음 연구 과제까지 제안하는 것

## 핵심 실험 루프

```
안건(연구 질문) → 템플릿 기반 환경 구성 → 테스트 진행 → 문서화(이미지 포함) → 결론 정리 → 다음 연구 과제 제안
```

예시 안건: *"레디스는 얼마의 요청이 들어올 때 블로킹이 걸릴까?"*

## 확정된 요구사항

### 1. 도메인 범위
- 컴퓨터/인프라 관련 전반 (예: Spring Boot, Redis(Docker), Kubernetes(Docker), ClickHouse 등)
- 특정 도메인에 고정되지 않고 계속 확장/추가됨 (초기 논의에서는 "데이터 과학 실험 랩"으로 시작했으나, 실제 예시들을 반영해 "컴퓨터/인프라 시스템 실험 전반"으로 범위 확정)

### 2. 실행 모델 (하이브리드 — 필수)
- **Claude Code 반자동 파이프라인**: 사용자가 안건을 주면 Claude Code가 템플릿 선택/환경 구성/테스트 실행/결과 분석/문서화까지 보조 수행
- **사람이 직접 실행 가능한 풀 스크립트**: Claude 의존성 없이 사람이 클론 후 바로 재현 가능한 실행 경로가 반드시 함께 존재해야 함

### 3. 재현성 & 기여 가능성 (필수 요건)
- 다른 사람이 프로젝트를 클론해서 실험을 재현하거나, 새로운 실험 템플릿을 기여할 수 있어야 함
- 따라서 템플릿 구조가 표준화되어 있어야 함 — 기술 스택이 달라도 기여자가 따라야 할 계약은 일관되어야 함

### 4. 발행(퍼블리싱)
- GitHub Pages, **Jekyll 네이티브** 사용 (별도 빌드 파이프라인 없이 markdown push만으로 배포 — 유지보수 부담 최소화가 이유)
- 각 실험 결과는 이미지(차트/그래프)를 포함한 블로그 포스트 형태로 정리됨

### 5. 첫 파일럿 실험
- 아직 미정. **템플릿/구조부터 먼저 완성**하기로 결정함
- "레디스 블로킹 임계치"는 개념 설명용 예시로 언급되었을 뿐, 확정된 파일럿 실험은 아님

### 6. 크로스플랫폼 재현성 (필수 요건)
- 특정 사용자 한 명의 환경이 아니라, **Mac / Windows / 그 외 PC 등 여러 사용자·여러 OS**에서 동일하게 재현 가능해야 함
- 공통 계약 중 `run.sh`(bash 스크립트)는 Windows 네이티브 환경에서 그대로 실행되지 않으므로, 이 요건과 충돌하지 않도록 실행 방식을 재검토해야 함 (예: Windows에서도 별도 설치 없이 동작하는 러너 선택, 또는 OS별 진입점 제공)

## 채택된 아키텍처 방향

브레인스토밍 중 아래 3가지 접근법을 비교했습니다.

| 접근법 | 설명 | 판단 |
|---|---|---|
| **A. Docker 기반 하이브리드 + Jekyll 네이티브** | Docker라는 공통 기판 위에서, 오케스트레이션 도구(docker-compose/kind/k3d 등)는 실험 성격에 맞게 자유. 대신 모든 템플릿이 공통 계약(아래)을 강제 | **채택** |
| B. Kubernetes 단일 기판 표준화 | 모든 실험을 kind/k3d 기반 K8s 위에서 통일 실행 | 기각 — 간단한 실험에도 오버헤드가 과함 |
| C. 완전 자유형 | 인프라 도구 강제 없이 문서(CONTRIBUTING.md)로만 규칙 안내 | 기각 — 재현성 요건, Claude 자동 파싱 요건과 충돌 |

### 채택된 방향(A)의 핵심 아이디어

- 모든 실험 템플릿은 **"Docker 위에서 동작한다"**는 원칙만 공통
- 실제 오케스트레이션 도구는 실험 성격에 맞게 자유 (예: Redis → docker-compose, K8s 실험 → kind/k3d, ClickHouse → 단일 컨테이너)
- 대신 아래 **3가지 공통 계약**을 모든 템플릿이 반드시 지켜야 함:
  1. `run.sh` — 표준 인자를 받는 실행 스크립트. **Claude 없이 사람이 바로 실행 가능**해야 함
  2. `results.json` / `.csv` — 표준 스키마의 원시 측정치 출력
  3. `README.template.md` — 안건 → 가설 → 구성 → 방법 → 결과(이미지) → 결론 → 다음 과제 순서의 문서 스켈레톤
- Claude Code 스킬은 이 계약 위에서 동작: 안건을 받아 적절한 템플릿을 고르거나 새로 생성 → `run.sh` 실행 → results 파싱 → 차트(이미지) 생성 → README 채우기 → Jekyll `_posts`에 발행

## 다음 모델(상세 설계 담당)이 결정해야 할 사항

이번 브레인스토밍에서 다루지 않았거나, 상세 설계 단계로 넘긴 항목입니다:

- 리포지토리 디렉토리 구조 구체안 (예: `templates/`, `experiments/`, `docs/`, `scripts/` 등의 정확한 배치)
- `results.json`/`.csv`의 정확한 스키마 정의
- 차트/이미지 생성 도구 표준 및 실행 방식
- Claude Code 스킬(`SKILL.md`)의 정확한 인터페이스와 각 단계 프롬프트 설계
- Jekyll 테마, `_config.yml`, 포스트 네이밍 규칙 등 사이트 세부 설정
- `CONTRIBUTING.md`에 담을 기여 가이드 상세 내용
- 새 기술 스택(예: Kafka, PostgreSQL 등) 템플릿 추가 시 검증 프로세스
- 실제 첫 파일럿 실험 선정 및 템플릿 검증
- **Windows/Mac/Linux 등 여러 OS에서 동일하게 재현 가능한 실행 방식** (`run.sh` 단일 계약이 Windows 네이티브와 충돌하는 문제의 구체적 해법 포함 — 예: Make/Task/Python CLI 등 크로스플랫폼 러너 채택 여부, WSL2 요구 여부, Docker Desktop을 공통 분모로 삼는 전략 등)

## 별도로 진행 중인 산출물

- **연구 주제(안건) 후보 리스트**: 8개 인프라 도메인(캐시, WAS/JVM, 오케스트레이션, OLAP, RDBMS, 메시징, 동시성/네트워크, 스토리지) 전문가 에이전트 병렬 브레인스토밍 + 취합 워크플로우로 별도 생성 중. 이 문서와는 독립적으로 관리됨.
